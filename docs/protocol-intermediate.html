1. Project Overview

Tour Planner is a full-stack web application for planning and logging bike, hike, running, and vacation tours. Users self-register, create tours, and record tour logs with statistics. Route data (distance, duration, map polyline) is fetched live from the OpenRouteService.org REST API and displayed on an interactive Leaflet map. The application runs in a three-container Docker Compose setup (PostgreSQL + ASP.NET Core backend + Nginx/React frontend).

This intermediate submission covers the full working integration: authentication, tour CRUD, tour log CRUD, map display, full-text search, and computed attributes (popularity, child-friendliness).

2.2 Architecture Decision: Repository Pattern

The Repository Pattern was chosen as the primary design pattern. A generic IRepository<T> base interface defines GetByIdAsync, GetAllAsync, AddAsync, UpdateAsync, DeleteAsync. Concrete implementations (TourRepository, TourLogRepository, UserRepository) extend this base and add domain-specific queries (e.g. GetByUserIdAsync, SearchAsync). This decouples the BL from EF Core and makes unit testing via mocks straightforward.

2.3 Architecture Decision: Layered Architecture

The backend is split into three projects mirroring the three layers:

  • TourPlanner.DAL – EF Core DbContext, entity classes, repository interfaces & implementations
  • TourPlanner.BL – Service interfaces & implementations, ORS HTTP client, DTOs, JWT/auth logic
  • TourPlanner.API – ASP.NET Core controllers, DI wiring, JWT middleware, CORS config

Dependency direction is strictly one-way: API → BL → DAL. No layer references a higher layer.

7. Unit Tests – Selection & Justification

The test suite contains 33 NUnit tests across two categories: service-layer tests (using Moq to mock repositories) and repository-layer tests (using EF Core InMemory database). This ensures both the business logic and the data access layer are covered independently.

7.1 Service Tests (via Moq)

Test Class# TestsWhat is testedWhy critical
AuthServiceTests 8 Register success; duplicate email; duplicate username; password too short; empty username; login success; wrong password; unknown email Authentication is the security boundary of the entire application. Bugs here allow account takeover or data leakage. BCrypt hashing, JWT generation, and input validation must all work correctly together.
TourServiceTests 7 Create with valid data; create with empty name (throws); update by owner; update by non-owner (throws UnauthorizedAccess); delete by non-owner; delete by owner; get tours; search returns matching tours TourService contains the core business logic including ownership checks. Without tests, a bug in the ownership guard could allow any user to modify or delete another user's tour data.
TourLogServiceTests 7 Create log success; difficulty out of range (1–5); rating out of range (1–5); get logs for tour; attempt to get logs for another user's tour; delete own log; delete non-owned log (forbidden) Tour logs store user-entered statistics used to compute popularity and child-friendliness. Invalid difficulty/rating values would corrupt computed results. Ownership tests prevent cross-user data access.
RouteServiceTests 3 Valid route returns distance + duration; geocoding failure returns null gracefully; directions failure returns null gracefully RouteService wraps the external ORS API. External APIs are unreliable (network timeouts, rate limits). Tests verify the service degrades gracefully so a failed ORS call does not crash tour creation.

7.2 Repository Tests (EF Core InMemory)

Test Class# TestsWhat is testedWhy critical
TourRepositoryTests 5 Add and retrieve by Id; get tours by userId (only own tours returned); full-text search by name; delete removes from DB; update persists changes The repository is the only point that touches the database. Bugs here (e.g. wrong WHERE clause on userId) would expose all users' tours. The search query is complex SQL and must be verified independently.
TourLogRepositoryTests 3 Add log and retrieve by tourId; logs of other tours not returned; delete removes only target log TourLogs are the source of computed attributes. Incorrect filtering would mix logs from different tours, corrupting popularity and child-friendliness values.
Test strategy: Service tests use Moq to isolate business logic from the database entirely — any failure points to the service code, not the ORM. Repository tests use EF Core InMemory to verify LINQ queries produce correct SQL semantics without requiring a real PostgreSQL instance in CI.
